## 1. 정기(자동) 파기 배치

DB 자동 파기는 Supabase **pg_cron** 으로 실행한다. 아래 표의 "실행 상태"는 2026-07-12 기준.

| 대상 | 보존/파기 주기 | 근거 | 실행(cron) | 상태 |
|---|---|---|---|---|
| refresh_tokens / rate_limits | 만료·폐기 후 즉시 | 인증 위생 | `auth-cleanup` 매시간(17분) | ✅ 운영 중 |
| 탈퇴자 격리 데이터(app.withdrawn_users) | 30일 후 | 재가입 악용 방지·최소보관 | `withdrawn-users-purge` 매일 03:43 | ✅ 운영 중 |
| 전화 인증코드(phone_verifications) | 발급 후 1일 | 데이터 최소화(코드 TTL 5분) | `retention-purge` 매일 03:23 | ✅ 운영 중 |
| 위치 인증 이력(location_verifications) | 6개월 후 | 내부 방침(사업계획서 §3.4) | `retention-purge` 매일 03:23 | ✅ 운영 중 |
| 사진 인증 로그(photo_verifications, 촬영 좌표 포함) | 6개월 후 | 위치정보 최소보관 | `retention-purge` 매일 03:23 | ✅ 운영 중 |
| 접속 로그(app.auth_logs)·게시글 조회기록(post_views, ip_hash) | 3개월 후 | 처리방침 §3 | `retention-purge` 매일 03:23 | ✅ 운영 중 |
| 위치정보 이용·제공사실 확인자료(app.location_usage_logs) | 6개월 보존 후 파기 | 위치정보법 제16조 제2항 | `retention-purge` 매일 03:23 | ✅ 운영 중 |
| 업체 인증 행 데이터(business_profiles) | 탈퇴 30일 / 반려 후 재신청 없음 6개월 | 처리방침 §3 | `retention-purge` 매일 03:23 | ✅ 운영 중 |
| 업체 인증 증빙 서류 파일(business-docs 버킷) | 반려 6개월 · 탈퇴 30일 · 교체 30일 (파기 큐 경유) | 처리방침 §3 | `business-docs-purge` 매일 04:13 → `purge-business-docs` 엣지 | ✅ 운영 중 |
| 관리자·취급자 접속기록 | **1년 이상 보관** | 개인정보 안전성 확보조치 기준 | (보관 의무 — 파기 아님) | — |

> `retention-purge` 배치는 pmdb 마이그레이션 `20260711060549_retention_purge_batch` ·
> `20260711065201_auth_logs` · `20260712033510_location_usage_logs` ·
> `20260715090000_business_rows_retention_purge`(업체 인증 행) 로 구현·적용 완료.
> 업체 인증 **서류 파일**은 스토리지 API 가 필요해 SQL 크론이 직접 지울 수 없다 —
> 파기 큐(app.business_doc_purge_queue)에 쌓인 due 행을 `purge-business-docs` 엣지
> 함수가 처리하며, pg_cron `business-docs-purge`(매일 04:13)가 pg_net 으로 호출한다
> (pmdb `20260715100000_business_docs_purge_cron`, 2026-07-15 연결·검증 완료 —
> retention-purge 03:23 이 행·큐를 정리한 뒤 50분 후 실행되는 순서).
> 구현 SQL 원본은 pmdb 저장소가 정본이다(아래 부록 A 는 초기 제안 스케치로, 실제 구현은
> FK 참조 행 스크럽 등 세부가 다르다).

### 위치정보 이용·제공사실 확인자료 (위치정보법 제16조 제2항)

- **저장소**: `app.location_usage_logs` — 약관 제8조②의 확인자료 항목(이용자 식별정보 ·
  이용·제공 일시 · 목적 · 제공받은 자)을 저장. 탈퇴 후에도 법정 6개월 분리 보존을 위해
  users FK 없이 보관하며, 6개월 경과분은 `retention-purge` 가 매일 파기.
- **자동 기록 지점**(AFTER INSERT 트리거): ① 활동지역 인증(location_verifications)
  ② 게시글 사진 촬영위치 검증(photo_verifications, 촬영 좌표 있는 행)
  ③ 게시글 작성 위치 기록(posts, 작성 좌표 있는 행). 도입 시점에 기존 이력도 소급 기록함.
- **기록 제외**: 시설 검색의 좌표 파라미터는 거리 정렬에만 일시 이용되고 저장·이용자
  식별 연계가 없어 확인자료 기록 대상에서 제외(근거는 마이그레이션 주석에 명시).
- **열람청구 대응**(약관 제10조): 고객센터로 열람 요구 접수 시 관리자가
  `admin_location_usage_logs` RPC 로 조회하여 지체 없이 회신.

### 백업과 파기의 관계
- **현재(Free 요금제)**: Supabase 자동 백업 미제공 → DB 레코드 삭제 시 **즉시 파기가 완결**.
  단, 수동 백업(pg_dump 등)을 보관하는 경우 그 파일에도 삭제 대상 데이터가 남으므로,
  **수동 백업은 짧은 보존주기(7일)로 회전·폐기**하고 그 사실을 기록한다.
  - 실행: 백엔드 저장소(pmdb) `scripts/backup.command`(더블클릭) 또는 `scripts/backup.sh`
    — public·app 스키마 덤프 + gpg 암호화 + 7일 회전·폐기 + 무결성 검증을 자동화.
- **정식 오픈 시(Pro 요금제로 업그레이드 예정)**: 자동 일일 백업 **보존 7일** → 삭제된
  개인·위치정보는 최대 7일 내 백업본에서도 소멸.

## 부록 A. 파기 배치 추가 마이그레이션(초기 제안 — 적용 완료, 이력 보존용)

> ✅ 2026-07-11~12 pmdb 마이그레이션으로 적용 완료. 실제 구현은 아래 스케치와 달리
> FK 참조 행의 좌표 스크럽, 접속 로그·확인자료 파기까지 포함한다 — **pmdb 저장소의
> `retention_purge_batch` / `auth_logs` / `location_usage_logs` 마이그레이션이 정본.**

```sql
-- 위치정보/인증코드 보존기간 경과분 자동 파기
create or replace function app.cleanup_retention()
returns void language sql security definer set search_path to '' as $$
  -- 전화 인증코드(OTP): 발급 1일 경과분 파기(코드 TTL 5분, 데이터 최소화)
  delete from public.phone_verifications  where created_at < now() - interval '1 day';
  -- 위치 인증 이력: 6개월 경과분 파기(사업계획서 §3.4)
  delete from public.location_verifications where created_at < now() - interval '6 months';
  -- 사진 인증 로그(촬영 좌표 포함): 6개월 경과분 파기
  delete from public.photo_verifications  where created_at < now() - interval '6 months';
$$;

-- 매일 03:23 실행
select cron.schedule('retention-purge', '23 3 * * *', $$select app.cleanup_retention();$$);
```

> 주의: `photo_verifications` 행 삭제는 스토리지 이미지 파일을 지우지 않는다(이미지는 게시글
> 수명과 함께 관리). 촬영 좌표 등 위치 메타데이터 로그만 파기된다.
